# Sanity Studio: инструкция владельцу Speaker One

Эта инструкция позволяет подключить и безопасно проверить админ-панель. Опубликованные изменения попадают на публичный сайт только после отдельной сборки и размещения нового `dist`.

## 1. Создайте аккаунт Sanity

1. Откройте [sanity.io](https://www.sanity.io/).
2. Нажмите **Get started** или **Log in**.
3. Войдите через предложенный провайдер учётной записи.
4. Включите MFA/двухфакторную защиту, если она доступна для выбранного способа входа.
5. Сохраните recovery-коды в защищённом хранилище, не в репозитории проекта.

## 2. Создайте проект

1. Откройте [manage.sanity.io](https://www.sanity.io/manage).
2. Выберите **Create project**.
3. Назовите проект `Speaker One`.
4. Владельцем проекта должна быть Ваша постоянная рабочая учётная запись.
5. Пока с CMS работает один владелец, не приглашайте дополнительных редакторов. Будущую роль выбирайте по [SANITY_ROLES.md](SANITY_ROLES.md) и возможностям текущего тарифа.

## 3. Получите Project ID

1. Откройте созданный проект в `manage.sanity.io`.
2. Перейдите в раздел с общими настройками проекта.
3. Найдите значение **Project ID** — короткую строку из латинских букв и цифр.
4. Скопируйте только Project ID. Это публичный идентификатор, не токен и не пароль.

## 4. Выберите dataset

Для первого production-контента используется имя:

```text
production
```

Проверьте раздел **Datasets** проекта:

- если `production` уже существует, используйте его;
- если его нет, после установки зависимостей выполните из папки `studio/`:

```powershell
npx.cmd sanity dataset create production
```

Проверьте visibility dataset в Sanity Manage. Если dataset private, будущая production-сборка должна получать данные через server/build read token из secret storage. Этот токен нельзя добавлять во frontend env или Studio.

Не создавайте несколько datasets с похожими именами без необходимости.

## 5. Заполните Studio env

Откройте PowerShell в корне проекта и выполните:

```powershell
Copy-Item -LiteralPath studio/.env.example -Destination studio/.env.local
```

Откройте `studio/.env.local` и заполните:

```dotenv
SANITY_STUDIO_PROJECT_ID=ВАШ_PROJECT_ID
SANITY_STUDIO_DATASET=production
SANITY_STUDIO_API_VERSION=2026-07-15
SANITY_STUDIO_USE_CDN=false
```

Не добавляйте кавычки и пробелы вокруг `=`.

## 6. При необходимости заполните public env

Публичный сайт на Stage 10B.4 использует эти значения только во время сборки. Создайте root `.env.local` отдельно, не копируя `studio/.env.local` и не удаляя существующий `VITE_LEAD_ENDPOINT`:

```dotenv
VITE_SANITY_PROJECT_ID=ВАШ_PROJECT_ID
VITE_SANITY_DATASET=production
VITE_SANITY_API_VERSION=2026-07-15
VITE_SANITY_USE_CDN=true
```

Не добавляйте `VITE_SANITY_TOKEN`: всё с префиксом `VITE_` видно посетителю сайта.

## 7. Установите зависимости Studio

Из корня проекта:

```powershell
Set-Location -LiteralPath studio
npm.cmd ci
```

Команда использует подготовленный `studio/package-lock.json`. Root-зависимости публичного сайта при этом не меняются.

## 8. Запустите Studio локально

Оставаясь в папке `studio/`, выполните:

```powershell
npm.cmd run dev
```

После запуска терминал покажет адрес. По умолчанию откройте:

[http://localhost:3333/](http://localhost:3333/)

Терминал нужно оставить открытым, пока Вы работаете в локальной Studio.

## 9. Войдите в админ-панель

1. На странице `localhost:3333` нажмите вход.
2. Используйте ту же учётную запись, которая владеет Sanity project.
3. Разрешите Studio работать с проектом Speaker One, если Sanity запросит подтверждение.
4. После входа Вы увидите русское меню всех разделов сайта: общие настройки, Hero, навигацию, философию, компетенции, эксперта, этапы, Speech Lab, форму, Footer, SEO, legal и медиа.

Если появляется сообщение об отсутствии доступа, проверьте Project ID, dataset и аккаунт владельца.

### Откройте визуальный редактор

Для визуального редактирования сначала запустите сайт из корня проекта:

```powershell
npm.cmd run dev:visual
```

Затем запустите Studio в отдельном окне терминала:

```powershell
Set-Location -LiteralPath studio
npm.cmd run dev
```

Откройте Studio по адресу `http://localhost:3333`, войдите в аккаунт и выберите **Редактор сайта**. Встроенный сайт открывается по адресу из `SANITY_STUDIO_PREVIEW_URL`; локальное значение по умолчанию — `http://localhost:5173`.

Для чтения Draft-версий preview-серверу нужен отдельный read token. Добавьте его только в корневой `.env.local`:

```dotenv
SANITY_API_READ_TOKEN=
```

Переменная не должна начинаться с `VITE_`: иначе секрет может попасть в клиентскую сборку. Не копируйте token в `studio/.env.local`, поля Sanity или Git. Подробная инструкция: [VISUAL_EDITING_GUIDE.md](VISUAL_EDITING_GUIDE.md).

## 10. Что сделать после первого входа

1. Включить MFA для учётной записи или провайдера, через который выполняется вход в Sanity.
2. Проверить Project ID в Sanity Manage и локальной конфигурации, не публикуя его в отчётах без необходимости.
3. Проверить, что Studio использует dataset `production`.
4. Проверить visibility dataset и зафиксировать public/private для будущей build-time интеграции.
5. Проверить список участников: сейчас должен остаться один владелец-Admin.
6. Не добавлять секреты в CMS, `.env.example`, скриншоты или Git.
7. Открыть каждый singleton-документ и убедиться, что основные разделы не показывают «Untitled».
8. Создать тестовый Draft без реальных секретов и персональных данных.
9. Проверить обязательные ошибки и warnings на слишком длинном тексте или неверном URL.
10. Опубликовать тестовый документ только после прохождения обязательной валидации.
11. Открыть **Редактор сайта**, нажать на тестовый текст и убедиться, что справа открывается соответствующее поле.
12. Проверить Desktop и Mobile в панели предпросмотра.
13. Убедиться, что Draft виден только в защищённом preview, а обычный публичный сайт показывает Published-контент.
14. Помнить, что Publish не заменяет утверждённый процесс сборки и размещения: SEO и итоговая production-версия обновляются только после новой сборки/deploy.
15. Удалить тестовые значения или позже заменить их утверждённым контентом. Не оставлять тестовые данные как production-контент.

Дополнительно откройте [SANITY_SECURITY.md](SANITY_SECURITY.md) и выполните контроль CORS по [SANITY_SETUP.md](SANITY_SETUP.md).

## 11. Создайте или перенесите содержательные документы

Сначала подготовьте медиа, потому что настройки и Hero ссылаются на изображения:

1. Откройте **Медиа**.
2. Нажмите создание нового документа.
3. Укажите понятное имя, назначение и alt-текст.
4. Загрузите изображение.
5. Подтвердите права на публикацию.
6. Нажмите **Publish**.

Затем откройте **Общие настройки**:

1. Заполните название и реальный HTTPS-домен.
2. Выберите заранее созданные логотипы и favicon.
3. Добавьте основной контакт и только рабочие социальные ссылки.
4. Нажмите **Publish**.

Singleton-документы не нужно создавать через общее меню: они открываются по своим постоянным пунктам и создаются при первом сохранении/публикации.

Чтобы подготовить Draft-копии утверждённого контента без ручного набора, сначала выполните безопасный просмотр плана:

```powershell
Set-Location -LiteralPath studio
npm.cmd run seed:dry
```

Перед первой записью выполните `npx sanity login`, затем `npm.cmd run seed:write`. Seed не публикует документы, но создаёт отсутствующие Draft и загружает текущие локальные изображения по стабильным ID. Если документ уже существует, обычный write сохранит его без перезаписи. Полная инструкция находится в [OWNER_ADMIN_GUIDE.md](OWNER_ADMIN_GUIDE.md).

После Publish вернитесь в корень проекта и выполните:

```powershell
npm.cmd run build
```

Скрипт загрузит только Published-данные и соберёт новый `dist`. До ручного размещения нового `dist` публичный домен не изменится. Подробная инструкция по полям и откату: [SANITY_EDITING_GUIDE.md](SANITY_EDITING_GUIDE.md).

## 12. Разверните Sanity Studio

После локальной проверки из папки `studio/` выполните:

```powershell
npm.cmd run deploy
```

При первом запуске Sanity предложит выбрать уникальный hostname. Используйте понятное имя без персональных данных, например:

```text
speaker-one-admin
```

Если имя свободно, Studio будет доступна примерно по адресу:

```text
https://speaker-one-admin.sanity.studio/
```

Фактический URL покажет CLI. Запишите его в защищённую внутреннюю документацию. Публикация Studio не делает её открытой для редактирования: вход по-прежнему требует аккаунт с доступом к Sanity project.

## 13. Как хранить токены безопасно

- Project ID, dataset и API version не являются секретами.
- Write token, read token для private dataset, webhook secret, Telegram bot token и CRM key являются секретами.
- Секреты хранятся только в secret settings хостинга/CI или защищённом password manager.
- Секрет нельзя отправлять в открытый чат, добавлять в screenshot или вставлять в Studio field.
- Для private dataset в будущем используйте отдельный server/build read-only token в secret storage. В текущем public build токена нет.
- При подозрении на утечку токен нужно немедленно revoke и создать заново.

## 14. Что нельзя публиковать в Git

Нельзя добавлять в repository:

- `studio/.env.local` и root `.env.local`;
- Sanity write/read tokens;
- пароли и recovery-коды;
- Telegram bot token и chat ID;
- CRM/API keys;
- webhook secrets;
- экспорты с персональными данными;
- screenshots, где виден секрет.

Перед передачей изменений проверяйте:

```powershell
git status --short
git diff --check
```

В Git должны попадать только `.env.example` с пустыми или безопасными публичными значениями.

## 15. Если Studio не запускается

Проверьте по порядку:

1. команда выполняется внутри `studio/`;
2. существует `studio/.env.local`;
3. `SANITY_STUDIO_PROJECT_ID` заполнен без пробелов;
4. dataset называется `production` и существует;
5. зависимости установлены через `npm.cmd ci`;
6. локальный порт 3333 не занят;
7. аккаунт имеет доступ к выбранному Sanity project.

Официальная документация: [установка Studio](https://www.sanity.io/docs/studio/installation), [environment variables](https://www.sanity.io/docs/studio/environment-variables), [deployment](https://www.sanity.io/docs/studio/deployment).
