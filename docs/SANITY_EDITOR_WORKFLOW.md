# Редакционный процесс Speaker One в Sanity

Публичный сайт пока работает только на `siteConfig.js`. Публикация документа в Sanity **не обновляет сайт автоматически**. Build-time подключение и пересборка будут добавлены на следующих этапах.

## Draft и Published

- **Draft** — сохранённые изменения, которые ещё не опубликованы. Их видит редактор с соответствующим доступом, но published-запрос публичного сайта их не получает.
- **Published** — последняя опубликованная версия документа в Content Lake.
- После редактирования уже опубликованного документа Sanity сохраняет новые изменения как Draft, а предыдущая Published-версия остаётся отдельно до следующей публикации.

## Как редактировать документ

1. Откройте нужный раздел слева. Критичные разделы открываются напрямую одним документом.
2. Измените только необходимые поля.
3. Прочитайте подсказки под полями.
4. Исправьте ошибки. Они блокируют публикацию, потому что данные обязательны или имеют неверный формат.
5. Проверьте warnings. Они не всегда блокируют публикацию, но предупреждают о длине текста, повторе ссылки или слишком большом изображении.
6. Оставьте изменения как Draft и перечитайте их.
7. На следующем этапе используйте Preview для desktop и mobile до публикации.
8. Нажимайте **Publish** только после проверки смысла, ссылок, изображений и предупреждений.
9. Помните: сейчас после Publish публичный сайт не изменится. Для этого позже будет отдельная проверяемая пересборка.

## Singleton-документы

В одном экземпляре существуют:

- «Общие настройки сайта»;
- «Первый экран»;
- «Навигация»;
- «SEO»;
- «Юридические данные».

Они открываются прямо из левого меню и имеют стабильные ID. Создать второй документ этого типа через обычное меню нельзя. Delete и Duplicate для них убраны; удаление критичных данных через внешние API всё равно должно быть запрещено организационными правилами и минимальными правами.

## Что проверять перед Publish

- нет ошибок в обязательных полях;
- предупреждения понятны и осознанно приняты;
- ссылки открываются и используют HTTPS, где требуется;
- изображение имеет понятное назначение и alt-текст;
- заголовки и описания не перегружены;
- ИНН и ОГРНИП введены строками без пробелов;
- в документе нет секретов или персональных данных, которые не предназначены для публикации.

## Что нельзя вводить в CMS

Не вводите токены, пароли, recovery-коды, bot token, chat ID, webhook secret, deploy token, CRM/API keys и серверные credentials. Если поле просит такие данные, остановитесь: в текущих схемах таких полей быть не должно.

## Отмена и история

- Неподходящий Draft можно отменить через штатное действие **Discard changes**.
- Для опубликованного документа используйте историю и сравнение изменений перед восстановлением версии.
- Массовое восстановление выполняется из проверенного экспорта dataset, а не вручную по памяти.
