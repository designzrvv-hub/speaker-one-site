# Безопасность Sanity для Speaker One

Статус: правила Stage 10B.2. Сейчас один владелец работает с проектом как Admin, публичный сайт не читает Sanity и не содержит Sanity-токенов.

## 1. Учётная запись владельца

- Используйте отдельную постоянную рабочую учётную запись, а не общий логин для нескольких людей.
- Включите MFA для способа входа, которым Вы пользуетесь в Sanity. Если вход выполняется через Google, GitHub или другой провайдер, MFA включается и проверяется у этого провайдера.
- Recovery-коды храните в менеджере паролей или другом защищённом хранилище вне репозитория.
- Не передавайте сессию браузера, пароль или MFA-код подрядчикам. Каждому будущему участнику создаётся отдельный доступ.

## 2. Участники и отзыв доступа

Проверяйте список участников в [Sanity Manage](https://www.sanity.io/manage): проект → **Settings** → **Members**.

- Сейчас в проекте должен оставаться только владелец с ролью Administrator.
- Перед приглашением редактора выберите минимально достаточную роль с учётом тарифа.
- После завершения работы сотрудника или подрядчика удалите его из проекта и организации, если доступ больше не нужен.
- После подозрительной активности завершите лишние сессии у провайдера входа, смените пароль, обновите MFA и отзовите затронутые токены.

## 3. Секреты

В Sanity-документах, схемах, browser env и Git запрещено хранить:

- write/read tokens;
- deploy tokens;
- webhook secrets;
- Telegram bot token и chat ID;
- CRM token;
- пароль;
- приватный API key;
- серверные credentials.

Project ID, dataset и API version являются идентификаторами конфигурации, но их всё равно не нужно выводить в публичные отчёты без причины. Серверные токены должны находиться только в secret storage CI/хостинга. `lead endpoint` остаётся в защищённых переменных окружения хостинга и не редактируется как контент Sanity.

## 4. Dataset и API

- Рабочий dataset называется `production`.
- Запись в dataset разрешается только авторизованным участникам и серверным процессам с минимальными правами.
- Не добавляйте write token во frontend и в переменные `VITE_*` или `SANITY_STUDIO_*`.
- Для browser CORS добавляйте только точные доверенные origins. Wildcard с credentials запрещён.
- Studio и будущий сайт должны работать только по HTTPS после публикации.

## 5. Резервное копирование и история

- Перед массовой миграцией, изменением схемы и первым production-подключением делайте экспорт dataset в защищённое хранилище.
- Автоматические Sanity Backups зависят от тарифа; не считайте их включёнными без проверки в Manage.
- Если автоматические backups недоступны, используйте регулярный `sanity datasets export production ...` и проверяйте возможность восстановления в отдельном тестовом dataset.
- Для отдельных документов используйте историю и сравнение изменений в Studio. История документа не заменяет полный экспорт dataset.
- Экспорты могут содержать черновики и персональные данные. Не добавляйте их в Git и ограничьте срок хранения.

## 6. Минимальный ежемесячный контроль

1. Проверить участников и роли.
2. Проверить MFA владельца и recovery-доступ.
3. Проверить CORS origins и удалить неиспользуемые.
4. Проверить API tokens и отозвать устаревшие.
5. Создать или проверить свежий экспорт `production`.
6. Просмотреть историю критичных singleton-документов.

Официальные справочные материалы: [роли](https://www.sanity.io/docs/user-guides/roles), [CORS](https://www.sanity.io/docs/content-lake/cors), [экспорт dataset](https://www.sanity.io/docs/cli-reference/cli-datasets), [backups](https://www.sanity.io/docs/content-lake/backups).
